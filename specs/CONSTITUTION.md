# Todo App – Phase I Constitution

**Date:** 2026-01-03  
**Version:** 1.0  
**Status:** Active  

## Purpose
This project is an AI-native, spec-driven Todo application built using Claude Code and Spec-Kit Plus. The Phase-I objective is to implement a clean, in-memory Python console application.

## Non-Negotiable Rules
1. **No manual code**: All code must be generated by Claude Code from specifications.
2. **Spec-first**: Any bug or change requires spec refinement — not direct code edits.
3. **In-memory storage**: Phase-I stores all data in memory only (no persistence).
4. **Console interaction**: stdin/stdout only; no GUI or web interface.

## Architectural Principles
- **Single source of truth**: Specifications drive all implementation.
- **Deterministic behavior**: Predictable, reproducible command execution.
- **Explicit command handling**: Clear command definitions and routing.
- **Separation of concerns**:
  - Models: Data structures and domain logic
  - Store: In-memory data management
  - Commands: Command definitions and execution
  - Application: Entrypoint and command dispatcher

## AI-Native Foundations (Forward-Compatible)
- **Command logic reusability**: Business logic must be usable by future AI agents and frameworks.
- **Framework-agnostic**: No hard-coded UI assumptions; business logic independent of presentation layer.
- **Modular design**: Clear interfaces between layers enable future refactoring (e.g., to web/API).

## Quality Bar
- **Clear function boundaries**: Each function has a single responsibility.
- **Type hints**: All functions must include type annotations.
- **Predictable console output**: Deterministic, parseable output format.
- **Defensive error handling**: Graceful handling of invalid input and edge cases.
- **Docstrings**: Clear documentation of function purpose and behavior.

## Language & Runtime
- **Python**: 3.13+ (type hints, modern stdlib)
- **Environment**: WSL2 / Linux
- **Dependencies**: Minimal external dependencies; prefer stdlib
- **Package structure**: Organized by architectural concerns

## Development Workflow
1. Write specification (`.md`) for feature/change
2. Claude Code generates all implementation from spec
3. Automated tests verify specification compliance
4. Commit with spec reference
5. Bugs → spec refinement, not code patches

## Phase-I Scope
- Core todo operations: create, read, update, delete, list
- Simple memory-based persistence
- Clean console UI (ASCII, human-readable)
- Command-driven interface
- Defensive error handling

## Phase-I Non-Scope
- Data persistence (file/database)
- Authentication/authorization
- Network/API
- Performance optimization
- Web/GUI interface
